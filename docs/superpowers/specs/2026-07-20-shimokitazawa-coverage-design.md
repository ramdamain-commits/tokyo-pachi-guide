# 下北沢エリア追加＋取材・イベント系強化 設計書

- 日付: 2026-07-20
- ステータス: 設計承認済み
- 関連: `2026-07-16-tokyo-pachi-guide-design.md`（初期設計）

## 目的

1. 対象エリアに下北沢を追加し、既存7エリア＋下北沢の8エリア体制にする
2. 取材・イベント系の情報を4方向で強化する
   - 店舗×媒体の取材実績の紐付け（静的データ）
   - 取材予定の確認手段（リンク集＋手順書）
   - 媒体マスタの拡充
   - ビューワの取材系表示強化

## データソース方針

取材実績・取材予定の調査は**相互検証の2段構え**とする。

- 集約サイト（スロカク・みんレポ等）で当たりを付ける
- 媒体公式スケジュール／結果ページで裏取りする
- 両者で確認できたものだけ `confidence: 高`、片側のみは `中` または `低`
- 全データに出典URLと確認日を付ける（stores.json と同思想）

収集委任時の制約（workspace 既存ルール踏襲）:

- researcher への一次抽出指示は curl 生HTML＋スクリプト機械抽出（WebFetch のAI要約は同一ページでも照会ごとに数値がぶれるため）。デコードは utf-8 → cp932 → shift_jis の順
- Playwright 必須サイト（bot遮断403等）は researcher に割り当てず、収集計画段階でメイン側作業として切り出す

## データ設計

### 案の比較と採用

| 案 | 内容 | 判断 |
|----|------|------|
| A（採用） | 新規 `data/coverage.json` に店舗ID×媒体名の紐付けを独立管理 | 特定日（ほぼ不変・確度高）と取材実績（変動が速い・伝聞ベース）を分離でき、四半期棚卸しの対象を絞れる |
| B | stores.json に media フィールド追加 | 確度の低い伝聞が確定マスタに混ざり棚卸しの見通しが悪化するため不採用 |
| C | media.json に storeIds を持たせる | 「この店に入る媒体は？」の主ユースケースと向きが逆のため不採用 |

### coverage.json スキーマ

```json
{
  "updated": "YYYY-MM-DD",
  "note": "過去の取材実績に基づく推定・伝聞情報であり、店舗・媒体の公式情報ではない",
  "coverage": [
    {
      "storeId": "pia-ueno",
      "mediaName": "スロパチステーション",
      "frequency": "月1回程度",
      "lastSeen": "2026-06",
      "confidence": "高",
      "sources": [ { "url": "...", "checkedAt": "YYYY-MM-DD" } ],
      "note": null
    }
  ]
}
```

- `storeId` は stores.json の `id`、`mediaName` は media.json の `name` と一致させる（結合キー）
- `confidence` は相互検証の結果で決める（上記データソース方針）
- 整合性（キーの参照先が存在すること）はテストで担保する

### media.json 拡張

- 来店系（いそまる・よしき等の来店イベント）・ホール系媒体を追加し、15〜20媒体規模に拡充
- 各媒体に `scheduleUrl`（公式取材スケジュールページURL）フィールドを追加。取材予定確認のリンク集の正データとなる

### stores.json

- 下北沢駅周辺徒歩圏の店舗（推定2〜4店）を既存スキーマのまま追加
- 調査手順は既存と同じ（集約サイト＋相互検証、特定日パターン・抽選・換金率・出典・確認日）

## フェーズ構成

| フェーズ | 内容 | 成果物 |
|---------|------|--------|
| 1 | 下北沢エリア追加 | stores.json 追記（コード変更は原則不要。エリア一覧がハードコードされていないか要確認） |
| 2 | 媒体マスタ拡充 | media.json（媒体追加＋scheduleUrl） |
| 3 | 店舗×媒体の紐付け | data/coverage.json 新設（全掲載店対象） |
| 4 | 取材予定の確認手段 | docs/取材予定の確認手順.md（巡回先リンク集＋読み方の手順書。データとしては持たない — 予定情報は日々変わるため） |
| 5 | ビューワ強化 | 取材傾向タブのクロス表示（媒体→実績店）、店舗詳細への「入りやすい媒体」バッジ、今日/明日タブへの取材実績バッジ、媒体の scheduleUrl 外部リンク |

フェーズ2→3は依存あり（mediaName の正規名がフェーズ2で確定）。フェーズ1とフェーズ2は独立して並行可能。

## ビューワ設計（フェーズ5）

- 結合ロジック（storeId/mediaName の突合・バッジ生成）は `match.mjs` 相当の純関数モジュールに切り出し、`node --test` でユニットテストを追加する
- 参照整合テスト: coverage.json の storeId が stores.json に、mediaName が media.json に必ず存在することを検証
- 既存5タブ構成は維持し、取材傾向タブを拡張する

## エラーハンドリング

- coverage.json が空・未取得でもビューワは既存表示のまま動作する（取材バッジが出ないだけ）
- 参照キー不整合はテストで検出し、実行時は該当エントリをスキップして console.warn

## 公開・運用面の注意

- 公開リポジトリのため、coverage.json と README に「過去実績に基づく推定・伝聞であり公式情報ではない」旨を明記する
- 収集データは店舗の公開情報・第三者サイトの推定のみ。個人情報・行動圏を特定しうる記述は含めない
- 台帳/TASKS 更新時は「再export→機微語grep→portal push」の順で回す（lifeops-portal.md）
- coverage.json は四半期棚卸しの対象に加える（stores.json と同じサイクル）

## テスト方針

- 既存 `test/match.test.mjs` は変更しない
- 新規: 結合ロジックの純関数テスト＋参照整合テスト
- 手動確認: ローカル `python3 -m http.server 8099` で5タブ表示、Pages公開後に公開URLで再確認
